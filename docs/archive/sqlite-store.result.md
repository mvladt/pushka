## Принятые решения (по факту реализации)

- **Схема:** ровно как в плане — `notifications(id PK, payload TEXT, datetime TEXT, subscription TEXT)` + индекс `idx_notifications_datetime`. Без лишних колонок.
- **Формат даты:** при записи `datetime` нормализуется в канонический UTC-ISO (`new Date(datetime).toISOString()`). Это делает лексикографическое сравнение TEXT в SQL хронологически корректным, несмотря на разнобой входных форматов (`"2025-01-01T00:00"` без таймзоны vs полный ISO с `Z`). Побочный эффект: хранимое/возвращаемое `datetime` теперь в UTC-ISO, а не байт-в-байт как прислал клиент. Момент времени тот же.
- **`getAllForNow`:** реализовано чистое окно `[now, now + 2 мин]` (SQL `WHERE datetime >= ? AND datetime <= ?`) — то самое намерение, что было в комментарии JSON-версии, без переноса её кривой логики (UTC-дата + локальные часы/минуты + асимметричное сравнение).
- **`removeMany`:** через `db.transaction(...)` — атомарно.
- **Соединение** не закрывается явно: живёт до конца процесса (как и допускал план).

## Замечен латентный баг (вне рамок задачи)

Окно `[now, now+2мин]` (как и старая JSON-логика) **не отправляет просроченные** уведомления: если процесс полежал и `datetime` ушёл в прошлое — уведомление никогда не попадёт в выборку и зависнет в базе навсегда. Стоит учесть при пункте todo «Рефакторинг scheduler-а» (вероятно, нижнюю границу окна убрать: `datetime <= now+2мин`).
