# Задача: резервное копирование notifications.db

## Проблема

На проде (`/srv/webpush-scheduler/data/notifications.db`) нет вообще никакого бэкапа. Проверено
на сервере: `/var/backups` содержит только служебные файлы `dpkg`/`apt`, ничего от приложений.
Если файл базы будет повреждён (диск, баг миграции — см. `sqlite-migrations.task.md`, риск №3,
неудачный деплой, ошибка администратора), восстанавливать неоткуда — теряются все запланированные
уведомления живых подписчиков.

## Желаемое поведение

- Периодический (например, ежедневный) снимок `notifications.db` через безопасный для живой базы
  метод — `sqlite3 notifications.db ".backup copy.db"` или `VACUUM INTO 'copy.db'`. Обычный `cp`
  не годится — рискует скопировать файл в процессе записи и получить битую копию.
- Снимки хранятся **не только на этом же сервере** — минимум одна копия должна пережить полную
  потерю диска/сервера (например, синхронизация на AMS или во внешнее хранилище).
- Ротация — не бесконечное накопление, а последние N копий (например, 7 ежедневных).
- Команда восстановления задокументирована в `deploy/README.md`.

## Риски

1. **Не путать с бэкапом перед миграцией** (`sqlite-migrations.task.md`, риск №3) — тот
   отдельный, разовый снимок непосредственно перед прогоном конкретной миграции. Эта задача —
   про регулярный процесс, не зависящий от деплоев и миграций.
2. **`dexity-server` на этом же сервере, судя по всему, в такой же ситуации** (тоже на
   `node:sqlite`, см. `server-management/CLAUDE.md`) — не проверялось, отдельная задача в
   репозитории `dexity`, вне скоупа этого проекта.
