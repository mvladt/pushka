# Задача: механизм миграций схемы SQLite

## Проблема

`createSqliteStore` (`src/sqliteStore/store.ts`) создаёт схему через
`CREATE TABLE IF NOT EXISTS` — это идемпотентно для *создания*, но ничего не делает,
если таблица уже существует с другой структурой. Механизма миграций в проекте нет вообще.

Это уже не гипотетический риск: `data/notifications.db` на проде живёт вне `releases/`
и симлинкуется в каждый релиз (см. `cicd.result.md`), то есть **переживает каждый деплой**.
Если в новом релизе поменяется схема (добавится колонка, поменяется тип, переименуется поле),
старая база на проде останется со старой структурой — приложение упадёт на первом же
`INSERT`/`SELECT`, либо (для nullable-полей) тихо потеряет данные.

При этом база больше не «исторические данные, о которых фиг с ним» (как было решено при
миграции JSON→SQLite, см. `docs/archive/sqlite-store.result.md`) — сейчас в ней лежат реальные
запланированные уведомления живых подписчиков. Потерять её при неудачной схеме — значит,
кто-то не получит push.

## Желаемое поведение

Лёгкий, без внешних зависимостей механизм версионирования схемы на базе
`PRAGMA user_version`:

- Список миграций в коде — упорядоченный массив `{ version, migrate(db) }` (или похожая
  структура), каждая миграция — обычный SQL (DDL и/или `UPDATE` для бэкфилла).
- При открытии БД (`createSqliteStore`) — прочитать `PRAGMA user_version`, применить по порядку
  все миграции с версией выше текущей, каждую в своей транзакции с обновлением
  `user_version` внутри той же транзакции (SQLite поддерживает транзакционный DDL) — если
  миграция упадёт посередине, файл базы останется нетронутым на предыдущей версии.
- **Миграция №1 — сама текущая схема**, то, что сейчас лежит в `CREATE TABLE IF NOT EXISTS`.
  На свежей базе создаёт таблицу и индекс, на проде — no-op (таблица уже есть), в обоих случаях
  после неё `user_version = 1`. Так текущий `CREATE TABLE IF NOT EXISTS` перестаёт быть отдельной
  веткой кода и становится частью общего списка миграций — один источник истины для схемы,
  а не два места, которые могут разойтись.
- Реальные изменения схемы (когда понадобятся) — миграции №2, №3... с обычным DDL.

## Принятые решения

- **Без библиотек** (drizzle-kit, umzug и т.п.) — оверкил для одной таблицы, лишняя
  зависимость. `PRAGMA user_version` + массив миграций в коде — тот же результат в десяток строк.
- **Только forward-only, без down-миграций.** Откат — восстановление бэкапа `data/notifications.db`
  + переключение `current` на предыдущий релиз (см. «Риски» про связь со схемой деплоя), не
  автоматический откат схемы.
- **Ограничение SQLite `ALTER TABLE`:** он поддерживает `ADD/RENAME/DROP COLUMN` и
  `RENAME TABLE`, но **не** смену типа/constraint колонки на месте — для этого нужен паттерн
  «новая таблица → `INSERT ... SELECT` → drop старой → rename». Аддитивные изменения (добавить
  колонку) — тривиальны через `ALTER`, для остального миграция явно делает пересборку таблицы.
- **API подтверждён на месте** (Node 22.18, тот же движок `node:sqlite`, что и на проде):
  ```js
  db.prepare("PRAGMA user_version").get()  // → { user_version: N }
  db.exec(`PRAGMA user_version = ${n}`)     // pragma не принимает bind-параметры
  ```
- Проверено по SSH: на проде (`/srv/webpush-scheduler/data/notifications.db`) сейчас
  `user_version = 0` — как и на любой свежей базе, конфликта с существующими данными нет,
  миграция №1 отработает как no-op.

## Риски

1. **Миграции выполняются при каждом старте сервиса**, включая обычный `systemctl restart`
   при деплое без изменений схемы (миграция уже применена → no-op, но код всё равно
   выполняется). Если миграция бросит исключение — сервис не поднимется. Митигация: гонять
   миграцию на копии прод-базы (или свежескачанной) перед деплоем в прод.
2. **Ломает текущую модель отката.** Сейчас откат — это просто переключить симлинк `current`
   на предыдущий релиз и рестартнуть (`cicd.plan.md`, этап 3.5), без пересборки. Если релиз
   принёс миграцию схемы, откат кода **не откатывает** уже смигрированную `data/notifications.db`
   — старый код может не пережить новую схему. Значит, откат релиза с миграцией требует ещё и
   восстановления бэкапа БД, это нужно явно прописать в `deploy/README.md` при реализации.
3. **Бэкап перед миграцией обязателен** — база больше не «данные, о которых фиг с ним» (см.
   «Проблема»). Механизм бэкапа `data/notifications.db` перед прогоном миграций — либо шаг
   деплоя, либо ручная команда в README, требует отдельного решения при реализации.
4. **Однопроцессный сервис** (systemd, один инстанс) — многопоточной/многопроцессной гонки на
   миграцию нет, это допущение, а не гарантия на будущее (если когда-то появится несколько
   инстансов — потребуется доработка).

## Тестирование (когда будет реализовываться)

- Тест на **свежей `:memory:` базе** — миграции применяются, финальный `user_version` верный
  (уже покрыт неявно текущими тестами `sqliteStore`, но стоит явно проверить `user_version`
  после инициализации).
- Тест на **базе со старой схемой** — вручную создать файл/`:memory:` с таблицей `notifications`
  без индекса и `user_version = 0` (или ниже целевого), прогнать инициализацию стора, убедиться,
  что (а) миграции применились, (б) **существующие строки не потеряны**. Это ключевое свойство —
  сохранность данных при апгрейде, а не просто «код не упал».
- Тест на **идемпотентность** — открыть один и тот же файл дважды подряд, второй раз миграции
  не должны выполняться повторно (или выполняться безопасно, если это по природе idempotent DDL).

## Когда делать

Не блокирует ничего прямо сейчас — `user_version = 0` является корректной ретроактивной базовой
точкой, ничего не потеряется, если отложить. Разумный момент для реализации — **первое реальное
изменение схемы `notifications`**: тогда этот механизм появляется вместе с конкретной миграцией
№2, а не как абстрактный скаффолдинг впрок.
